请求转发和响应重定向

web应用间接访问项目资源（静态和动态）

请求转发：a->b  b->c
重定向： a->b  a->c


请求转发：客户端发送一个请求报文，由 Tomcat 将其封装为 resquest && response 对象
        根据 servlet-1 匹配，将其交给一个 service 方法，结果该 servlet-1 的 service 在判断相关参数后发现自己处理不了这个请求
        编写代码将这个请求 resquest && response 对象发送给其他 servlet-2，由这个 servlet-2 的 service 方法处理

        <上述行为中一个只有一对 resquest && response 对象，是服务器内部的行为，客户端是不知道的>
        1.客户端只发送了一次请求，接收一次响应
        2.请求转发时，请求和响应对象会被传递给下一个对象
        3.请求中的参数可以继续向下传递
        4.转发是服务器内部的行为，客户端是不知道的


重定向: 客户端发送一个请求报文，由 Tomcat 将其封装为 resquest && response 对象
       根据 servlet-1 匹配，将其交给一个 service 方法，结果该 servlet-1 的 service 在判断相关参数后发现自己处理不了这个请求
       编写代码在这个请求中的 reponse 对象中设置状态码：302，并且给其响应头中添加一个响应头 Location: “目标资源”
       客户端收到响应以后会自动根据响应中的 location 中声明的目标资源，对其进行请求
       客户端再次向目标资源（Location 指向）发送请求报文，由 Tomcat 将其封装为 resquest && response 对象



同样实现页面跳转，优先使用重定向，在重定向解决不了的时候（如需要参数继续传递的时候，需要请求 WEB-INF内的资源）再考虑使用请求转发
请求转发有点过于屏蔽真实请求路径，有些路径是和真实路径有关（有些时候静态资源下载失败）
